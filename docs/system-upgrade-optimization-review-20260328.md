## 4. 逐条建议的修订说明

## 4.1 收紧接口防护与请求资源控制

### 修订后的判断

不需要把这项当成当前必须做的高优先级改造。

### 当前状态

当前阶段已完成其中唯一值得顺手做的子项：

- 已把全局请求体限制从原先的 `50mb` 收紧为默认 `10mb`
- 并保留 `CLASSMANAGER_REQUEST_BODY_LIMIT` 环境变量以便未来按需调整

### 为什么前一版会把它提得更高

前一版文档是按一个更保守的威胁模型写的。那个模型默认认为：

- 服务可能被更多同网段设备频繁访问
- 接口未来可能逐步开放
- 登录与维护口令需要预防常规内网攻击行为

在那个模型下，限流、锁定、访问异常告警是合理的高优先级。

### 为什么在当前场景下可以降级

按你补充的真实使用前提，这个系统当前：

- 不对公网开放
- 没有明显的对抗性使用场景
- 使用范围较小
- 不属于“开放给陌生人使用”的后台

在这个前提下，下面这些改造都可以延期：

- 登录限流
- 注册限流
- 维护密码连续失败锁定
- 401/403/429 异常频率告警
- 为防攻击而专门做的网络层加固

这些事情不是没价值，而是“当前投入产出比不高”。

### 这项里还保留什么价值

虽然整套防攻击方案现在可以降级，但其中有一个子项仍然有一点现实价值：

- 适度收紧全局请求体大小

这件事的意义不主要是防恶意攻击，而是防误操作和防前端异常。

例如当前全局 `express.json()` 和 `express.urlencoded()` 允许到 `50mb`，但业务自己的单次保存上限实际只有 `8MB`。这说明解析器的入口边界比业务边界宽得多。

如果哪天前端同步逻辑误把一个超大的对象、缓存块或异常状态一并发上来，即使不是恶意，也会先给 Node 增加不必要的内存和解析负担。

### 建议做法

当前不建议上整套接口防护，而建议只保留一个低成本动作：

- 把全局 body limit 适度收紧

可以考虑：

- 全局先降到 `10mb` 左右
- 真有特殊接口需要更大，再局部放宽

### 如果现在完全不做，风险是什么

在当前环境下，这项不做的风险主要不是“被攻击”，而是：

- 某次前端 bug 把异常大对象发给服务器
- 某次状态同步把不该发的大块内容一起发了出去
- 服务不会马上被攻破，但会变慢、卡顿或更吃内存

### 具体情境举例

情境：

某次前端同步代码改动后，误把大体积临时缓存对象拼进了保存请求。因为解析器入口给到了 `50mb`，服务会老老实实把这团数据先读进来再处理。最终未必写库成功，但服务器已经做了无意义的大对象解析。

## 4.2 处理 `xlsx` 依赖的高危安全债

### 修订后的判断

不建议现在为了安全告警单独发起一次大改，也不建议立刻替换整套 Excel 导入方案。

### 当前状态

当前阶段已完成低成本加固，但没有换库，也没有改掉现有 Excel 功能：

- 已新增统一导入守卫
- 已增加扩展名校验
- 已增加文件大小限制
- 已增加工作表数量限制
- 已增加按场景区分的最大记录数限制
- 已接入学生名单、励志语录、考试成绩、藏宝阁四个导入入口

### 为什么前一版把它提得更高

前一版强调的是：

- `xlsx` 当前确实有 `high` 告警
- 它不是闲置依赖，而是真正参与用户导入链路
- 处理外部文件的组件原则上都应该更谨慎

这个判断本身没有错。

### 为什么在当前场景下可以降级

你补充了两个关键事实：

1. `xlsx` 导入是必要功能，不能砍
2. 导入文件的来源整体可信，不需要按高度对抗性场景设计

在这个前提下，当前更合理的结论是：

- 这是一个需要记录的稳定性与依赖债
- 但不是一个现在必须立刻重构的安全救火项

也就是说，`npm audit` 报告依然成立，但处理优先级应该下降。

### 当前还剩下什么现实风险

就目前系统形态看，`xlsx` 的现实风险更像：

- 导入某份异常文件后，浏览器页面卡住
- 某个工作表结构异常，导致导入后的业务对象不稳定
- 某次解析异常让前端状态脏掉

而不是“服务器正在被恶意 Excel 攻击”。

### 建议做法

当前不建议大动，但建议保留几个低成本加固项：

- 限制导入文件大小
- 限制可读取的最大行数
- 限制可读取的工作表数量
- 导入前给出更明确的异常提示
- 出错时尽量不要把异常解析结果直接并入主状态

这些动作不改变功能，也不要求你换库，风险较低。

### 如果现在不做，风险是什么

风险主要集中在可用性而非对抗性安全：

- 某份格式异常的 Excel 可能让标签页明显卡顿
- 导入边界条件不严时，状态结构可能变得难排查
- 真实用户感知会是“导入不好用”或“导入后界面异常”

### 具体情境举例

情境：

老师拿到一份从别的系统导出的 Excel，结构并不完全符合当前模板。文件不是恶意文件，但工作表列数很多、空列很多、格式很乱。导入时浏览器明显变卡，最后弹出的错误也不够清楚。这时问题更像稳定性和产品体验问题，而不是攻击问题。

### 何时需要重新提高优先级

如果以后出现下面任一情况，就应重新提升这项优先级：

- Excel 文件来源开始变得不可控
- 系统开放给更大范围的用户上传文件
- 导入链路频繁出现场景性异常
- 计划顺手做依赖治理窗口

## 4.3 停止在“读数据”时自动删除未知数据域

### 修订后的判断

这项目前不再列为高优先级改造，可以暂时观察，不必马上动。

### 当前状态

当前阶段未改动这项逻辑，仍按“未来设计债”处理。

### 为什么前一版会把它列为风险

从设计原理上说，“读取接口带删除副作用”确实比“读取只读取”更脆弱。

它的典型风险场景是：

- 新旧版本同时存在
- 多设备混跑不同版本
- 经常回滚
- 新数据域迭代频繁

在这些场景里，旧版本读取新数据时可能把新字段顺手删掉。

### 为什么当前可以降级

你补充了一个非常关键的现实前提：

- 当前基本不会出现旧浏览器标签页长时间不刷新地继续工作

再结合当前系统形态：

- 访问方式单一
- 客户端形态单一
- 多版本长期并存的概率低

那么这项风险就从“当前兼容性隐患”下降成“未来设计债”。

### 关于“这不是本来就是你加进去的吗”

是的，这类逻辑通常是为了做兼容清理或历史字段收口才加进去的。它在当时的目标下可能是合理的，并不等于“当时就错了”。

但“当时合理”和“长期最稳”不是一回事。

当前更准确的结论应该是：

- 这段逻辑不代表现在马上会出事
- 但如果以后系统开始频繁扩字段、做版本回滚、或出现多端混用，它会成为需要回头处理的设计点

### 建议做法

当前可以不改。

但建议保留一条未来原则：

- 如果后续继续新增大量数据域，再考虑把“读时清理”改成“显式迁移脚本”

### 如果现在不做，风险是什么

在当前使用前提下，不做的现实风险较低。

真正的风险主要出现在未来这些情况：

- 同账号在两台机器上运行不同版本页面
- 回滚到旧版本后继续使用很久
- 引入越来越多新数据域

### 具体情境举例

情境：

如果未来新增了一个新数据域，而你又恰好在另一台机器上长期留着旧版本页面不刷新使用，那么旧版本读取数据时可能顺手把新域清掉。但按你现在的实际使用方式，这种情境并不常见，所以这项暂不需要急着处理。

## 4.4 处理 `path-to-regexp` 安全告警

### 修订后的判断

这项当前可以明确延期，不值得为了它单独开一轮升级。

### 当前状态

当前阶段未处理，继续延期。

### 为什么前一版会提到它

原因很直接：

- `npm audit --omit=dev` 确实报了 `high`
- 它位于 Express 路由匹配链路
- 这类问题理论上可能导致拒绝服务型卡顿

这些判断都成立。

### 为什么当前不值得优先处理

按你现在的实际场景：

- 系统不对公网开放
- 就算未来放开，也只会开放很少一部分接口
- 当前局域网环境中，刻意构造路由攻击的现实概率很低

在这个前提下，为了一个当前低现实风险的告警去单独动依赖，并不划算。

### 这项升级本身有没有风险

有，但风险不算大，只是没必要现在承担。

原因是：

- 路由匹配链路升级后，理论上仍可能出现边缘行为变化

当前建议：

- 先记录
- 延期处理
- 等以后本来就要做依赖维护，或系统暴露面明显扩大时再一起处理

### 如果现在不做，风险是什么

当前不做的风险主要还是理论风险，不是现实高频问题。

### 具体情境举例

情境：

如果以后真的把更多查询接口开放给更大范围设备访问，那么这类路由匹配告警的优先级就会提升。因为到那时它面对的就不再是一个很小的受控使用面，而是更广的调用面。

## 4.5 补齐 `SIGTERM` 优雅停机

### 修订后的判断

这项值得现在做，而且是当前最明确、最稳妥的改造项。

### 当前状态

当前阶段已完成：

- 已抽出统一停机路径
- 已同时处理 `SIGINT` 与 `SIGTERM`
- 已加入超时强退保护
- 已通过测试实际验证 `SIGTERM` 下可正常退出

### 原理

命令行里手动停服务，常见信号是 `SIGINT`。但 `systemd` 在停止或重启服务时，默认发的是 `SIGTERM`。

当前服务已经通过 `systemd --user` 运行，但代码里只处理了 `SIGINT`，没有处理 `SIGTERM`。这意味着：

- 手工 `Ctrl+C` 的退出路径考虑到了
- `systemd restart` 的退出路径没有完整考虑

### 为什么这项现在特别值得做

它有几个特点：

- 与是否公网无关
- 与是否有人攻击无关
- 与是否更换依赖无关
- 改动面小
- 收益直接作用在你当前已经在用的运行方式上

这是典型的低风险、高确定性收益改造。

### 建议做法

建议把停机逻辑抽成统一函数，例如：

- 一个 `shutdown()` 负责关闭 server、清理 timer、关闭 SQLite
- 同时监听 `SIGINT` 和 `SIGTERM`
- 给关闭过程加超时保护，避免卡死

### 如果不做，会有什么风险

风险不是“系统必坏”，而是：

- `systemctl --user restart` 时，请求可能在处理中被中断
- 部署切换时保存请求可能失败
- 将来如果后台任务更多，停机不完整会更明显

### 具体情境举例

情境：

你执行 `systemctl --user restart classmanager-app.service` 发布新版本。此时刚好有人正在保存数据。没有 `SIGTERM` 优雅停机时，请求可能被中途打断，前端只会看到“保存失败”，但问题根源其实是停机路径不完整。

## 4.6 建立 CI、lint、类型约束和更细粒度测试

### 修订后的判断

不建议现在上完整工程化体系，但仍建议保留最小化验证护栏。

### 当前状态

当前阶段已完成最小化增强，但没有上完整工程体系：

- 保留了现有 `npm test`
- 扩充了现有 smoke test
- 新增了测试会话下的考勤主流程验证
- 新增了 `SIGTERM` 优雅停机验证

以下内容仍未启动，且当前不急：

- 完整 CI 平台化
- 很重的 ESLint 规则
- 全面 TypeScript 化
- 大规模测试工程重构

### 为什么这项不能简单等于“完全不需要”

你现在虽然是单人开发，而且大量依赖 AI 工具，但这并不意味着验证体系没有意义。

恰恰相反，AI 写代码的特点是：

- 改动速度快
- 改动面可能比人手动修改更大
- 局部需求解决了，但容易顺手影响别的链路

所以你未必需要很重的工程体系，但你仍然需要“最低限度的自动验证”。

### 为什么不建议现在上全套

按当前团队形态：

- 不需要为了多人协作上完整 PR 流水线
- 不需要为了代码规范统一去推很重的 lint 规则
- 不需要为了类型收益立刻把项目推进到全面 TypeScript

这些事情都可能增加维护负担。

### 当前真正建议做的，是最小化版本

建议保留或补齐的最小护栏：

1. 继续保留并使用 `npm test`
2. 再补少量关键回归测试
3. 形成固定发布前检查动作

优先补的测试可以是：

- 登录与鉴权
- 维护密码
- 数据保存与读取
- 一条考勤主流程

### 当前可以先不做的部分

以下内容当前都可以延期：

- 完整 CI 平台化
- 很重的 ESLint 规则
- 全面 TypeScript 化
- 大规模测试工程重构

### 如果现在完全不做，会有什么风险

风险主要不是多人协作冲突，而是：

- AI 帮你改完一个需求时，顺手把别的链路改坏
- 你自己回头看改动时，很难保证所有核心流程都还正常
- 后续做小改动会越来越依赖手工点页面验证

### 具体情境举例

情境：

你让 AI 去修某个数据同步问题，结果它顺手改了保存逻辑。主界面看起来正常，但维护权限判断或考勤保存被影响了。如果没有最小化测试护栏，这种问题通常只能靠你手工逐页回归才能发现。

## 5. 当前不应优先做的大版本升级

以下升级当前仍然不应抢在前面做：

- React 19
- Express 5
- Tailwind 4
- better-sqlite3 12

### 原理

这些升级带来的主要不是“当前痛点立刻消失”，而是：

- 行为差异
- 回归验证成本
- 与现有功能耦合后的不确定性

在你当前场景下，先把运维停机路径和最小化验证做好，比大版本迁移更划算。

### 建议做法

- 大版本升级放到单独窗口处理
- 不和业务改动混在一起
- 只有在有明确收益时再启动

## 6. 修订后的推荐执行顺序

当前阶段已完成：

1. 补齐 `SIGTERM` 优雅停机
2. 保留并补强最小化测试护栏
3. 收紧全局 body limit
4. 为 `xlsx` 导入补上轻量级保护

剩余建议顺序如下：

1. 持续观察“读时删未知域”是否真的带来兼容问题
2. 等依赖治理窗口再看 `path-to-regexp`
3. 更重的接口安全防护继续延期
4. 完整 CI、lint、TypeScript 与大规模测试体系继续延期

## 7. 哪些变化会让优先级重新上升

如果未来出现下面任一变化，这份文档里的延期项就要重新提升优先级：

- 系统开始对公网或更大范围网络开放
- 同账号经常在多台设备、多版本页面之间切换
- Excel 文件来源变得不再可信
- 开发者数量增加
- 开始频繁做版本回滚
- 计划集中做一轮依赖治理

## 8. 一句话总结

基于当前真实环境，本轮真正值得做的 4 项已经完成：

- `SIGTERM` 优雅停机
- 最小化测试护栏增强
- 全局请求体限制收紧
- `xlsx` 导入轻量保护

当前剩下的大多数问题都更像“继续观察或延期处理的设计债”，而不是“必须马上处理的现实风险”。
