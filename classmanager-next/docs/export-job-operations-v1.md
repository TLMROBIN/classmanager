# 导出任务运维策略 v1

## 1. 目的

本文档用于定义 `export_jobs` 在 Phase A 阶段的运维规则。

当前重点不是复杂调度，而是把以下几件事定清楚：

- 状态如何解释
- 文件保留多久
- 什么时候标记过期
- 失败任务怎么处理
- `manifest` 与任务表如何协同

## 2. 当前状态定义

### 2.1 `queued`

含义：

- 任务已创建
- 尚未进入执行

### 2.2 `running`

含义：

- 任务已开始执行
- 结果文件可能尚未生成

### 2.3 `succeeded`

含义：

- 任务执行完成
- 输出文件已生成
- `summary`、`outputPath`、`finishedAt` 已写入

### 2.4 `failed`

含义：

- 任务执行失败
- `errorMessage` 应有值

### 2.5 `expired`

含义：

- 任务历史仍保留
- 但导出文件已视为过期
- 文件可能已删除，也可能仅逻辑过期

## 3. 当前保留策略

Phase A 当前默认策略：

- 成功导出后写入 `expiresAt`
- 当前保留期：`7 天`

说明：

- 这是 Phase A 的保守默认值
- 后续进入对象存储或正式任务中心后可调整为租户级配置

## 4. 当前清理策略

当前脚本：

- `npm run export:cleanup`

行为分 3 档：

### 4.1 Dry-run

默认行为：

- 只扫描
- 不改数据库
- 不删文件
- 不改 `manifest`

### 4.2 逻辑过期

命令：

- `npm run export:cleanup -- --apply`

行为：

- 把已过期任务标记为 `expired`
- 更新 `manifest`
- 不删除文件
- 为每个过期任务写入 `export.cleanup` 审计

### 4.3 物理清理

命令：

- `npm run export:cleanup -- --apply --remove-files`

行为：

- 把已过期任务标记为 `expired`
- 删除对应导出文件
- 更新 `manifest`
- 为每个过期任务写入 `export.cleanup` 审计

## 5. 失败任务策略

当前规则：

- 失败任务保留
- 不自动重试
- 通过 `errorMessage` 定位失败原因

当前明确不做：

- 自动重试队列
- 指数退避
- 死信队列

这些属于 Phase B 之后的能力。

## 6. manifest 协同规则

当前约束：

- `export_jobs` 是主任务记录
- `manifest.json` 是文件清单兼容层

当前要求：

- 成功任务应写入 `export_jobs`
- 成功文件应写入 `manifest`
- 清理过期文件时必须同步更新 `manifest`

禁止：

- 只删文件不改 `manifest`
- 只改 `manifest` 不改任务状态

## 7. 人工处理建议

### 7.1 发现大量 `failed`

先检查：

- API 日志
- `errorMessage`
- 输出目录权限
- PostgreSQL 连接状态

### 7.2 发现 `manifest` 与任务表不一致

优先以：

- `export_jobs`

作为主记录来源。

`manifest` 可重建，不应视为唯一事实源。

### 7.3 发现文件已丢失但任务仍是 `succeeded`

处理原则：

- 不改历史审计
- 可在后续增加“文件缺失”修复任务
- 不建议直接硬改为 `failed`

## 8. 当前下载授权规则

当前规则：

- 创建结构化导出任务需要具备导出写权限
- 直接结构化导出下载需要具备导出写权限
- 任务文件下载仅允许任务创建者本人，或具备导出写权限的成员
- 已标记 `expired` 的任务返回过期，不再按“文件不可用”处理
- 导出摘要与历史会同时展示 `export.structured` 和 `export.cleanup` 审计

## 9. 当前边界

Phase A 运维策略只适用于：

- 本地或单实例环境
- 同步导出执行
- 本地文件系统存储

不适用于：

- 多实例部署
- 对象存储
- 分布式 worker

## 10. 当前结论

导出任务线在 Phase A 已具备：

- 状态记录
- 文件保留期
- 干跑清理
- 逻辑过期
- 物理清理

下一阶段要做的不是继续堆脚本，而是进入：

- Phase B 的任务执行解耦
- 下载授权
- 更正式的保留与恢复策略
