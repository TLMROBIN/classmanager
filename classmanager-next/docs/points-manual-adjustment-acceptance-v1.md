# 积分手工调整验收与约束 v1

## 1. 文档目的

本文档用于定义新系统当前已实现的“单人手工积分调整 + 批量调整/修正/撤销”能力的验收范围、操作约束和后续扩展边界。

这份文档的目的不是设计完整积分系统，而是防止后续在尚未稳定前直接扩展到复杂审批和更多入口。

## 2. 当前已实现范围

当前已经落地的能力如下：

### 2.1 后端接口

- `POST /api/classes/:classId/points/adjustments`
- `POST /api/classes/:classId/points/batch-adjustments`
- `GET /api/classes/:classId/points/batch-adjustments`
- `POST /api/classes/:classId/points/batch-adjustments/revert`
- `POST /api/classes/:classId/points/batch-adjustments/:batchId/correct`
- `POST /api/classes/:classId/points/batch-adjustments/:batchId/revert`
- `POST /api/classes/:classId/points/transactions/:transactionId/revert`
- `POST /api/classes/:classId/points/audits/:auditId/revert`

### 2.2 前端入口

- 学生详情页中的“单人加减分”表单
- 学生列表页中的“批量加减分”表单
- 学生列表页中的“撤销最近一次批量加减分”入口
- 学生列表页中的“历史批量调整”列表与修正、撤销入口
- 学生列表页中的“历史批量调整”按类型、日期范围筛选，并支持理由/场景/类别/人数/分值/批次搜索
- 学生列表页中的“最近积分操作”审计列表与统一撤销入口
- 最近积分操作支持按单条/批量动作、可撤销状态、日期范围筛选，并按学生/理由/操作人搜索
- 最近积分操作与历史批量调整支持当前筛选结果摘要
- 最近手工调整的“撤销”按钮
- 学生详情最近手工调整会优先按审计 ID 走统一撤销入口
- 学生详情单条积分提交确认会展示类型、理由、场景、类别与余额变化预览
- 学生详情单条积分撤销确认会展示类型、分值、理由、场景、类别、来源和发生时间
- 学生列表批量积分调整 / 修正 / 撤销确认会展示影响人数、总分变化与学生预览
- 最近一次写入/撤销结果反馈
- 预设理由选择

### 2.3 已验证行为

- 可对单个学生手工加分
- 可对单个学生手工扣分
- 可对当前勾选的多名学生做同理由批量加分/扣分
- 可从历史批量调整载入修正模式，并以“撤销旧批次 + 新建替代批次”的方式完成修正
- 可撤销前端刚完成的最近一次批量积分调整
- 可撤销历史批量积分调整
- 可按审计 ID 撤销单条手工积分调整，以及批量积分调整中的单条记录
- 可在学生列表中查看最近积分审计操作并直接撤销
- 写入后会同步更新：
  - `point_transactions`
  - `point_accounts`
  - `audit_logs`
- 仅手工调整流水允许撤销
- 撤销后：
  - 原流水标记为已撤销
  - 新增反向流水
  - 账户数据回退
  - 写入审计日志

## 3. 当前验收标准

### 3.1 写入成功标准

满足以下全部条件才算通过：

1. 请求参数完整且合法
2. 目标学生属于当前班级
3. 写入成功后新增一条积分流水
4. 账户余额与累计积分按预期变化
5. 写入一条审计日志
6. 学生详情页刷新后能看到最新流水

### 3.2 撤销成功标准

满足以下全部条件才算通过：

1. 只能撤销 `manual_adjustment` 或任意 `manual_batch_adjustment`
2. 已撤销流水不可重复撤销
3. 撤销时新增一条反向流水
4. 原流水写入 `isReverted = true`
5. 账户数值回退到预期状态
6. 写入一条撤销审计日志
7. 前端列表能显示“已撤销”

## 4. 当前操作约束

### 4.1 当前批量范围是最小版

当前阶段只允许：

- 单个学生
- 单笔调整
- 明确理由
- 或多名学生同理由批量写入
- 批量仅支持 `bonus / penalty`

当前明确不做：

- 不同学生不同理由的批量写入
- 跨模块自动批量写分

当前已做但范围受限：

- 历史批量修正 / 撤销仅覆盖带 `batchId` 的新批次
- 单条手工调整已统一到审计撤销入口，批量历史仍保留 `batchId` 撤销入口

### 4.2 只支持手工调整场景

当前写接口只面向：

- 老师或管理员在学生详情页做即时手工调整

当前不承担：

- 考勤自动派生分数
- 作业登记自动写分
- 任务完成自动写分

这些能力后续要走各自模块服务，不应直接复用手工调整入口作为主流程。

### 4.3 撤销范围受限

当前只允许撤销：

- `sourceModule = manual_adjustment`
- `sourceModule = manual_batch_adjustment`

当前不允许撤销：

- 导入历史流水
- 自动计算产生的流水
- 未来的跨模块业务流水

## 5. 当前已知限制

### 5.1 权限已进入第一阶段

当前已校验：

- 当前用户属于该班级
- 若成员已分配角色，则只有具备积分管理权限的角色可以写入或撤销
- 若迁移样本尚未分配角色，则按兼容策略暂时放行，避免阻断现阶段验证

当前尚未细化：

- 谁可以撤销谁的操作
- 是否按班主任 / 任课教师 / 年级管理员进一步细分
- 不同租户套餐下的权限差异

### 5.2 表单仍是最小版

当前只具备：

- 理由
- 类型
- 分值
- 场景
- 类别
- 预设模板

尚未具备：

- 更细粒度的二次确认策略
- 原因备注富文本
- 最近常用理由
- 操作人和时间维度筛选

### 5.3 批量能力仍然受限

当前仍不能替代旧系统中的：

- 工资发放
- 作业批量登记
- 不同学生不同理由的复杂批量写入

## 6. 后续扩展门槛

在没有满足以下条件前，不建议扩展到更复杂的批量写能力：

1. 单人写入链路稳定
2. 撤销链路稳定
3. 权限模型补齐
4. 审计字段确认完整
5. 前端错误反馈和确认流程补齐

## 7. 推荐下一步

按风险从低到高，建议顺序如下：

1. 把角色分配和默认角色初始化正式接入迁移流程
2. 再评估是否进入更复杂模板化批量写入和更细粒度历史治理

## 8. 当前判断

当前“积分手工调整 + 批量修正 / 撤销”已经具备最小可用性，可以作为新系统第一个真实写操作闭环。

但它还不是旧系统积分操作模块的完整替代。要成为真正可切换能力，仍需后续补齐：

- 权限
- 批量
- 模块化自动写分
- 更完善的操作反馈与限制
