# 导出文件下载与可见性策略 v1

## 1. 目的

本文档用于定义 `export_jobs` 进入下载阶段前的权限和可见性规则。

当前已经落下最小下载实现，并锁定了这些边界：

- 谁能看到哪些导出任务
- 谁能下载哪些文件
- 任务过期后应如何处理
- 本地文件路径能否直接暴露

## 2. 当前风险

Phase A 目前已经有：

- `outputPath`
- 本地文件落盘
- 任务查询接口

如果后续直接暴露下载，而不先定义规则，会出现这些问题：

- 用户看到不属于自己的导出
- 本地绝对路径直接暴露到前端
- 过期文件仍可被下载
- 没有区分“可见”和“可下载”

## 3. 角色与权限原则

### 3.1 当前最低原则

在进入正式下载前，默认要求：

- 只有能访问该班级的成员才能看到该班级导出任务

### 3.2 下载权限建议

建议默认与“导出创建权限”等级保持一致：

- `tenant_owner`
- `tenant_admin`
- `class_admin`
- `class_teacher`

后续如果要细分，可增加：

- `export_manager`

当前不建议让普通只读成员直接下载结构化导出。

## 4. 可见性分层

建议把导出任务分成两层可见性：

### 4.1 任务元信息可见

包括：

- 导出类型
- 时间范围
- 状态
- 创建时间
- 创建人

### 4.2 文件下载可见

包括：

- 文件下载入口
- 文件名
- 文件大小
- 下载令牌 / 下载地址

规则：

- 能看任务，不代表能下载文件

## 5. 关于 `outputPath`

当前 `outputPath` 仅适用于：

- 本地开发
- 运维排错
- 脚本调试

不应作为前端长期公开字段。

进入下载阶段后建议：

- 前端列表不要直接展示绝对路径
- API 返回逻辑文件名或下载句柄
- 真正下载由单独接口处理

## 6. 过期文件规则

### 6.1 已过期任务

状态为：

- `expired`

规则：

- 可以继续保留任务记录
- 不应再提供下载
- 可允许查看历史元信息

### 6.2 文件缺失

如果任务是 `succeeded` 但文件不存在：

- 不自动改任务状态
- 查询接口可增加 `fileAvailable=false`
- 下载接口返回明确错误

## 7. 当前下载接口

当前已实现：

- `GET /api/export-jobs/:jobId/download`

当前行为：

1. 校验登录态
2. 校验班级访问权限
3. 校验任务归属或管理权限
4. 校验任务状态为 `succeeded`
5. 校验未过期
6. 校验文件路径仍在受控导出目录内
7. 校验文件存在
8. 返回文件流

当前前端行为：

- 列表与历史不再暴露本地绝对路径，只返回逻辑文件名和 `fileAvailable`
- 后端统一返回 `fileState`，用于表达 `pending / available / expired / missing / invalid_path / unavailable / manifest_only`
- 任务列表会展示创建人摘要与当前不可下载原因
- 只对 `succeeded`、`fileAvailable=true` 且当前用户有下载资格的任务显示下载按钮
- 已过期任务在前端直接禁用下载入口

## 8. 审计建议

下载也应单独审计，不应只依赖“创建导出任务”的审计。

建议动作名：

- `export.download`

至少记录：

- 谁下载
- 下载哪个任务
- 下载时间
- 文件类型
- 文件范围

## 9. 当前结论

当前实现仍遵守以下原则：

- `export_jobs` 可见性先按班级权限控制
- 下载权限不自动等于查看权限
- 不把 `outputPath` 当成前端公开字段
- 文件可用性由后端 `fileState` 统一表达
- `expired` 任务保留元信息，不再开放下载

后续如果继续扩下载能力，优先补：

1. 更细的下载权限模型
2. 创建人/下载人更明确的前端展示
3. 下载任务前端更明确的状态展示
