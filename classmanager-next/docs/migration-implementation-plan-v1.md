# 迁移实施计划 v1

## 1. 文档目的

本文档用于把系统重构从“设计阶段”推进到“执行阶段”。它回答以下问题：

- 旧系统如何在不中断使用的前提下迁移
- 新系统应按什么顺序建设
- 每一期交付什么、验收什么、如何回滚
- 哪些风险必须提前控制

本文档默认采用“双轨迁移”方案：

- 旧系统继续运行
- 新系统独立建设
- 分模块迁移
- 灰度验证
- 最后切换

## 2. 总体迁移策略

### 2.1 迁移原则

- 不做一次性重写后硬切
- 不在旧系统上继续叠加大功能
- 先锁定模型，再做代码
- 先迁核心事实数据，再迁玩法型模块
- 所有迁移都必须可追踪、可重跑、可核对

### 2.2 双轨运行定义

双轨运行分为 4 个状态：

1. 旧系统主写，新系统开发中
2. 旧系统主写，新系统灰度读
3. 旧系统主写，新系统灰度写或双写
4. 新系统主写，旧系统只读归档

### 2.3 切换底线

未满足以下条件前，不允许切主：

- 核心 P0 功能全部可用
- 旧数据迁移演练通过
- 至少一轮真实班级灰度通过
- 核心报错、审计、备份能力可用
- 关键指标达到预期

## 3. 项目组织建议

### 3.1 工作流分线

建议按 4 条工作线并行推进：

- A 线：产品与领域建模
- B 线：新系统底座与基础工程
- C 线：数据迁移与校验
- D 线：旧系统托底治理

### 3.2 每条线的职责

#### A 线：产品与领域建模

负责：

- 功能边界确认
- 名词统一
- 规则确认
- 验收清单确认

#### B 线：新系统底座与基础工程

负责：

- 新仓初始化
- 前后端脚手架
- Prisma schema
- 鉴权、租户、审计、日志

#### C 线：数据迁移与校验

负责：

- 旧数据导出
- 映射转换
- 导入流程
- 差异校验
- 演练报告

#### D 线：旧系统托底治理

负责：

- 冻结高风险改动
- 增加备份
- 修安全问题
- 控制生产风险

## 4. 阶段划分

## 4.1 第 0 期：冻结与托底治理

目标：

- 防止旧系统在重构期间继续恶化
- 建立最基本的数据安全与操作边界

执行项：

1. 冻结旧系统大功能新增
2. 禁止直接在 `public/script.js` 继续叠加大模块
3. 建立数据库每日备份
4. 导出当前所有用户业务数据快照
5. 记录当前生产数据版本
6. 修复高风险安全项

建议立即处理的高风险项：

- 默认管理员弱口令
- 默认 JWT 密钥
- `/api/adjustments` 这类可能失效的遗留接口
- 关键导入操作缺少保护

交付物：

- 当前仓库文档基线
- 数据备份策略说明
- 风险清单
- 旧系统变更冻结规则

完成标准：

- 每日备份可验证恢复
- 旧系统再无无规划大改
- 当前数据可完整导出

回滚方式：

- 不涉及系统切换，无需业务回滚

## 4.2 第 1 期：新系统基础工程落地

目标：

- 建立新系统的可开发底座

执行项：

1. 创建新仓或新 monorepo
2. 建立 `web`、`api`、`admin` 基础应用
3. 配置 TypeScript、ESLint、Prettier、测试框架
4. 建立 PostgreSQL 与 Prisma
5. 落地最小 schema
6. 建立用户认证与租户体系
7. 建立审计日志中间件
8. 建立基础 CI

推荐目录：

```text
classmanager-next/
  apps/
    web/
    api/
    admin/
  packages/
    ui/
    types/
    config/
  prisma/
  docs/
```

交付物：

- 新仓初始化完成
- 本地开发环境可启动
- PostgreSQL 可迁移
- 首个登录接口与健康检查接口可用

完成标准：

- `web`、`api`、`admin` 可以本地启动
- Prisma migration 可执行
- JWT/refresh token 工作正常
- 审计日志可落库

回滚方式：

- 该阶段不影响旧系统，无需生产回滚

## 4.3 第 2 期：P0 模块开发与迁移打样

目标：

- 打通第一批最小业务闭环

范围：

- 用户与租户
- 班级与学生
- 组织结构
- 积分流水
- 考勤
- 基础配置

执行项：

1. 完成对应 API
2. 完成最小前端页面
3. 编写旧数据导出脚本
4. 编写旧数据转换脚本
5. 编写新库导入脚本
6. 编写迁移映射核验脚本

交付物：

- P0 模块可用
- 针对一个样本用户的完整迁移演练
- 差异校验报告

完成标准：

- 学生数量一致
- 学生组织关系一致
- 积分余额与历史核对通过
- 考勤总量与关键样本核对通过
- 配置迁移后功能可运行

回滚方式：

- 新系统演练失败时，直接丢弃目标库重建
- 旧系统仍然主写，不受影响

## 4.4 第 3 期：P1 模块迁移

目标：

- 补齐高价值增强模块

范围：

- 作业
- 任务
- 商城/藏宝阁
- 管理后台
- 快照/导入导出

执行项：

1. 设计第二阶段 schema
2. 编写对应 API 与页面
3. 补充迁移脚本
4. 完成操作日志与库存事务设计

交付物：

- P1 模块上线灰度版
- 后台管理初版
- 导入导出工具初版

完成标准：

- 作业、任务、商城逻辑可独立运行
- 核心事务具备幂等与回滚保护
- 运营管理操作可审计

回滚方式：

- 按模块关闭 feature flag
- 仍以旧系统作为主写入口

## 4.5 第 4 期：灰度与双写验证

目标：

- 让真实用户在可控范围内使用新系统

灰度策略：

- 先选 1 个测试班级或 1 个真实低风险用户
- 先只读验证
- 再局部写入验证
- 最后扩大灰度范围

建议顺序：

1. 只读灰度
2. 单模块写灰度
3. 多模块写灰度
4. 全量灰度

执行项：

1. 配置灰度用户白名单
2. 开启只读比对
3. 开启局部双写或双算
4. 记录差异和错误
5. 每日复盘修正

交付物：

- 灰度运行日报
- 差异问题清单
- 修复清单

完成标准：

- 连续一段观察期内无重大数据偏差
- 用户反馈通过
- 无阻断级线上问题

回滚方式：

- 一键切回旧系统入口
- 停用新系统写入
- 保留灰度期新数据用于排查

## 4.6 第 5 期：正式切换

目标：

- 新系统成为唯一主入口

执行项：

1. 停止旧系统写入
2. 进行最终增量迁移
3. 核对关键数据
4. 切换域名或入口
5. 开启生产监控和告警

交付物：

- 新系统正式上线
- 旧系统归档说明
- 上线报告

完成标准：

- 用户全部从新入口访问
- 旧系统切换为只读
- 关键指标稳定

回滚方式：

- 如果在切换窗口内发现阻断问题：
  - 停止新写入
  - 恢复旧入口
  - 重新执行最终增量迁移

## 5. 数据迁移流程

## 5.1 单次迁移标准流程

1. 从旧系统导出原始数据
2. 生成中间标准化文件
3. 运行字段校验与映射校验
4. 导入到新库
5. 执行一致性检查
6. 生成人工抽检报告

### 5.2 导出文件建议

建议以用户/班级为单位导出：

- `users.json`
- `class.json`
- `students.json`
- `history.json`
- `attendance.json`
- `config.json`
- `treasures.json`
- `tasks.json`
- `battle.json`

### 5.3 中间标准化文件建议

中间层不要直接保留旧结构，建议输出：

- `tenant.normalized.json`
- `students.normalized.json`
- `point-transactions.normalized.json`
- `attendance-records.normalized.json`
- `configs.normalized.json`

### 5.4 迁移脚本分层

建议拆成 3 类脚本：

- `extract-*`: 从旧库提取
- `transform-*`: 清洗和映射
- `load-*`: 写入新库

命名示例：

- `extract-legacy-user-data.ts`
- `transform-legacy-students.ts`
- `load-point-transactions.ts`

## 6. 一致性校验方案

### 6.1 必查指标

- 用户数
- 班级数
- 学生数
- 小组数
- 宿舍数
- 岗位任命数
- 积分流水条数
- 每名学生的余额、累计积分、惩罚值
- 考勤记录条数

### 6.2 抽检规则

每次迁移演练至少人工抽检：

- 3 名学生的完整历史
- 3 天考勤记录
- 3 种积分模板
- 3 个库存/兑换样本

### 6.3 差异处理等级

- S1：数据丢失、金额/积分错误、考勤错乱
- S2：配置偏差、统计不一致、富文本展示异常
- S3：样式、排序、文案问题

规则：

- 存在 S1 不允许灰度
- 存在 S2 可继续修复后复测
- S3 不阻断迁移演练

## 7. 代码实施顺序

建议先后顺序如下：

1. 新仓初始化
2. Prisma schema
3. auth/tenant 模块
4. class/student 模块
5. point 模块
6. attendance 模块
7. 数据迁移脚本
8. web 端 P0 页面
9. admin 基础台
10. P1 模块

原因：

- 先立边界，再立事实数据，再补页面

## 8. 人员与任务拆分建议

如果由 1 人主导：

- 先完成文档、schema、迁移脚本
- 再做 API
- 最后做前端

如果由多人协作：

- 一人主导 schema 与迁移
- 一人主导 API
- 一人主导 web 页面
- 一人主导 admin 与导入导出

## 9. 风险清单与控制措施

### 9.1 最大风险

1. 旧数据字段不一致
2. 旧系统仍在变化导致迁移基线漂移
3. 新旧积分口径不一致
4. 考勤规则迁移后行为改变
5. 商城/库存事务设计不当

### 9.2 控制措施

- 旧系统冻结大改
- 每次演练前固定导出版本
- 建迁移映射表
- 建差异校验脚本
- 先做单租户单班级演练
- 关键操作必须事务化

## 10. Feature Flag 策略

新系统所有非基础模块必须可开关：

- `attendance`
- `homework`
- `tasks`
- `shop`
- `battle`
- `reports`
- `admin_tools`

用途：

- 灰度
- 回滚
- 套餐控制

## 11. 验收标准

### 11.1 第 1 阶段验收

- 新仓工程可启动
- 登录与租户链路可跑通
- PostgreSQL migration 正常

### 11.2 第 2 阶段验收

- 学生、积分、考勤最小闭环可用
- 单用户迁移演练通过

### 11.3 第 3 阶段验收

- P1 模块可灰度使用
- 导入导出能力可用

### 11.4 正式切换验收

- 用户可在新系统完成日常主流程
- 数据核对通过
- 监控、备份、审计到位

## 12. 当前建议的立即行动

基于当前文档状态，下一步建议按以下顺序继续执行：

1. 创建新系统目录或新仓
2. 初始化 monorepo
3. 建立 Prisma schema 初版
4. 创建 PostgreSQL 本地开发环境
5. 编写旧系统导出脚本

## 13. 本文档结论

当前已经具备开始编码的条件，但正确的起点不是先写页面，而是：

- 先建新工程
- 先定 schema
- 先写迁移脚本
- 先跑一个样本迁移闭环

## 14. 下一步

下一步最合适的是直接开始工程落地。建议顺序二选一：

1. 我继续在当前目录下创建新系统骨架与基础配置
2. 我先补一份 `Prisma schema 初版` 文档/代码草稿，再开始建仓

从执行效率看，推荐直接开始建新系统骨架。
