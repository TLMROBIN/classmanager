# 配置分层方案 v1

## 1. 文档目的

本文档用于约束新系统中的配置边界，避免重构后再次回到“所有配置堆在一个大对象里、谁都能改”的状态。

目标：

- 明确系统配置与模块配置的职责边界
- 降低跨模块修改带来的回归风险
- 为后续配置页拆分、权限控制、导入导出和审计提供基础

## 2. 核心原则

### 2.1 两层配置

新系统中的配置只分两层：

1. 系统配置  
   被多个模块共享、影响全局运行方式的配置。

2. 模块配置  
   只被单一业务域理解和消费的配置。

### 2.2 判断规则

满足以下任一条件，归入系统配置：

- 两个及以上模块都会读
- 影响班级或租户的全局行为
- 属于通用字典、组织结构或治理规则
- 需要被统一导出、统一审计、统一权限控制

满足以下条件，归入模块配置：

- 只有一个业务域理解其语义
- 修改后只应影响该模块
- 配置字段名称和规则明显带有模块上下文

### 2.3 禁止事项

- 模块直接写别的模块的配置
- 前端直接拼写跨模块配置 JSON
- 用单个 `extra` / `metadata` 大对象长期承载主配置
- 在“设置页”里把所有模块规则混成一个编辑入口

## 3. 系统配置范围

系统配置建议放在 `class_configs`、`feature_flags` 和少量基础字典表中。

### 3.1 班级基础配置

适合放入系统配置：

- 班级名称
- 时区
- 冻结状态
- 倒计时事件
- 课程备注

### 3.2 组织结构字典

适合放入系统配置：

- 小组定义
- 宿舍定义
- 岗位定义

原因：

- 多个模块都会引用这些结构
- 它们不是某一个模块私有的规则

### 3.3 通用积分模板

适合放入系统配置：

- 通用加减分模板
- 模板的场景、类别、基础分值

说明：

- 模板本身属于班级级共享资源
- 具体模块如何调用模板，仍由模块服务决定

### 3.4 功能开关

适合放入系统配置：

- 模块启用/停用
- 租户级功能开关
- 班级级功能开关

## 4. 模块配置范围

### 4.1 考勤模块配置

建议独立管理：

- 考勤时段
- 迟到线
- 周末规则
- 特殊日期规则
- 全勤奖励和迟到/缺勤扣分策略

原因：

- 只有考勤模块理解这些字段的实际含义

### 4.2 作业模块配置

建议独立管理：

- 学科列表
- 课代表配置
- 作业登记规则
- 作业周期规则

### 4.3 任务模块配置

建议独立管理：

- 任务默认奖励
- 发布规则
- 时间窗口策略

### 4.4 宝物/商城模块配置

建议独立管理：

- 商品定义
- 库存规则
- 兑换限制
- 使用规则

### 4.5 对战模块配置

建议独立管理：

- 赛季规则
- 队伍规则
- 结算策略
- 日志规则

## 5. 配置读写约束

### 5.1 后端约束

- 所有配置写入必须经 service 层
- 配置变更必须写审计日志
- 配置对象必须 schema 校验
- 配置读取 API 按“系统 / 模块”分组输出

### 5.2 前端约束

- 设置页先展示系统配置总览
- 模块配置进入各自模块页面维护
- 不再保留一个超级“设置页”承担所有编辑职责

## 6. 导入导出约束

- 系统配置导出单独成包
- 模块配置按模块导出
- 导入时必须声明配置包类型与版本
- 禁止再做“整班级所有配置混成一个 JSON 覆盖导入”

## 7. 当前落地建议

### 7.1 先进入系统配置的内容

- 班级基础配置
- 功能开关
- 小组 / 宿舍 / 岗位
- 通用积分模板

### 7.2 暂留模块配置的内容

- 考勤策略与时段
- 作业学科与课代表
- 任务规则
- 宝物规则
- 对战规则

## 8. 实施顺序

1. 保持当前数据库结构中系统配置与模块配置的读接口分离
2. 前端先把“设置”页限定为系统配置总览
3. 后续模块页分别接入自己的配置面板
4. 等配置边界稳定后，再开放编辑能力
