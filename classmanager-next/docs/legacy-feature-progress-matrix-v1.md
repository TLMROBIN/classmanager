# 旧功能实现进度对照表 v1

## 1. 文档目的

本文档用于把“旧系统已有功能”与“新系统当前实现状态”放到同一张表里，避免后续推进时凭印象判断。

用途：

- 明确哪些旧能力已经在新系统中有只读闭环
- 明确哪些功能只是部分完成
- 明确哪些模块目前因为样本为空或尚未建模而未开始
- 为后续排期、验收和切换提供统一基线
- 阶段性人工回归可配合 `docs/upgrade-regression-checklist-v1.md`

## 2. 状态定义

- `已实现-只读`：新系统已经可读取真实数据并在新前端展示
- `部分实现`：只完成了部分数据面、只完成了接口或仅完成底座
- `未开始`：尚未在新系统中实现
- `暂缓`：旧系统有功能，但当前样本无真实数据，不适合先做

## 5. 当前仍在兼容承接态的模块

以下模块已经在新系统中可用，但当前仍主要挂在“旧功能兼容区”，尚未拆成正式领域模型：

- 任务系统
- 藏宝阁 / 商城 / 祈愿
- 对战系统 / 双子星

对这些模块的建议策略：

- 先保证旧数据和旧流程可继续使用
- 优先补高频缺口，不急着重做成新模块
- 等真实使用稳定后，再决定是否正式拆域迁移

## 6. 当前最缺的能力

虽然只读面已经覆盖核心数据，但以下能力仍是切换前必须补的：

### 6.1 写操作

- 剩余高频写操作的治理细节
- 导出治理与权限继续收口

### 6.2 管理能力

- 用户与租户治理继续收口
- 权限细化
- 后台更完整的运维工具与真实数据验收

### 6.3 导出与恢复能力

- 更完整的文件治理
- 二次迁移与灾备恢复演练
- 导入导出任务中心继续收口

## 7. 下一步建议

按当前状态，后续应遵循以下顺序：

1. 先补核心模块剩余写操作闭环
2. 再收口后台治理、权限和运维工具
3. 再把旧功能兼容区逐步拆成更稳定的正式模块

## 8. 当前判断

新系统已经完成了“核心教学管理数据面”的主体只读实现，尤其是：

- 学生
- 积分
- 考勤
- 作业
- 设置

这意味着当前最大缺口已经不再是“看不到数据”，而是“还需要把剩余高频写操作、兼容区收口和恢复治理继续补齐，才能更稳地接近正式切换状态”。
