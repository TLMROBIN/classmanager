# 现有功能清单 v1

## 1. 文档目的

本文档用于梳理当前旧系统已经存在的功能、状态、依赖与迁移价值，作为新系统重构时的功能盘点基线。

本文档的目标不是复述 UI，而是回答以下问题：

- 当前系统到底已经实现了哪些功能
- 每个功能依赖哪些数据
- 哪些功能属于核心能力，必须迁移
- 哪些功能属于扩展能力，可后移
- 哪些功能带有高耦合实现，需要重构而不是直接照搬

## 2. 功能分级规则

### P0：必须迁移

直接影响系统基本可用性和主要业务价值：

- 登录认证
- 班级/学生档案
- 积分与流水
- 考勤
- 基础配置
- 数据导入导出

### P1：应迁移

有较高使用价值，但不必成为第一批切换阻塞项：

- 作业登记
- 任务系统
- 商城/藏宝阁
- 管理后台
- 快照与恢复

### P2：可后移

属于增强能力、玩法能力或运维辅助能力：

- 双子星对战
- 测试模式
- 桌面端探测/模式切换
- 自动惩罚衰减等策略性增强

## 3. 系统入口功能

### 3.1 登录与注册

入口：

- `public/login.html`
- `/api/auth/register`
- `/api/auth/login`

已实现能力：

- 用户注册
- 用户登录
- 登录成功后写入本地 token 和 user
- 登录前清理旧本地缓存
- 注册后自动登录

依赖数据：

- `users` 表
- 本地存储 `token`
- 本地存储 `user`

问题判断：

- 属于 P0 必迁功能
- 当前只有基础认证，没有密码重置、租户邀请、refresh token
- 当前本地缓存清理逻辑散落在页面脚本中，迁移时应统一到认证层

### 3.2 主应用入口

入口：

- `public/index.html`
- `public/script.js`

已实现能力：

- 登录校验
- 注入鉴权 header
- 401 自动登出
- 读取当前用户与管理员身份

问题判断：

- 属于 P0 必迁
- 当前鉴权逻辑通过全局函数挂在 `window` 上，需改为前端 auth module

### 3.3 管理后台入口

入口：

- `public/admin.html`
- `/api/admin/users`
- `/api/admin/users/:id/role`
- `/api/admin/users/:id`
- `/api/admin/stats`

已实现能力：

- 查看用户列表
- 查看用户数据量
- 修改用户角色
- 删除用户
- 查看注册用户总数和数据记录总数

问题判断：

- 属于 P1
- 当前是单页轻后台，后续应迁到独立管理台或后台模块
- 当前统计能力极弱，仅适合作为运营辅助页面

## 4. 主业务模块

### 4.1 仪表盘

入口：

- `Nav.dashboard`
- `DashboardView`

已实现能力：

- 展示学生相关概览
- 展示积分与历史统计类信息
- 值日安排设置
- 专员/心理委员配置
- 部分操作支持撤销

依赖状态：

- `students`
- `history`
- `config`

迁移判断：

- 属于 P0
- 新系统中应拆为“班级总览”“组织配置”“快速操作入口”三个子页面

### 4.2 积分管理

入口：

- `Nav.operations`
- `OperationView`
- `updatePoints`
- `batchUpdatePoints`
- `handleWage`
- `handleUndo`

已实现能力：

- 单人/多人加分扣分
- 按小组、宿舍筛选
- 使用预设理由加减分
- 自定义理由
- 工资发放
- 作业未交扣分登记
- 历史记录撤销

依赖状态：

- `students`
- `history`
- `config`
- 预设理由来自系统配置

迁移判断：

- 属于 P0 核心能力
- 这是当前系统最核心的业务模块之一
- 新系统必须改为“积分账户 + 流水”模型，不可继续前端直接改余额

### 4.3 考勤模块

入口：

- `Nav.attendance`
- `AttendanceModule`

已实现能力：

- 多时段考勤
- 学生签到
- 根据时段和规则判断迟到/缺勤
- 教师留言
- 学生留言
- 批量修正
- 批量缺勤处理
- 缺勤结算
- 全勤奖励
- 撤销由考勤产生的积分变动
- 按日期、时段、类型查询考勤情况

依赖状态：

- `attendanceRecords`
- `students`
- `history`
- `config`
- `messages`
- `teacherMessages`
- `quotes`

迁移判断：

- 属于 P0
- 需要拆成“原始考勤记录”“考勤规则”“积分派生结果”三个层次
- 当前模块耦合留言、语录、实时状态判断，迁移时要拆分职责

### 4.4 任务系统

入口：

- `Nav.tasks`
- `TasksView`
- `handleClaimTask`

已实现能力：

- 发布限时任务
- 设置开始时间、结束时间、积分奖励
- 学生领取任务
- 限制一项任务只能被一个人领取
- 管理员删除任务
- 重置任务领取状态

依赖状态：

- `tasks`
- `students`
- `history`

迁移判断：

- 属于 P1
- 功能价值明确，但当前实现比较轻量，后续可升级为“任务发布/领取/完成/审核”模型

### 4.5 藏宝阁 / 商城

入口：

- `Nav.treasure`
- `TreasureView`
- `handleRedeemTreasure`
- `handleReturnItem`
- `handleUseItem`

已实现能力：

- 商品展示
- 商品编辑
- 兑换商品
- 记录兑换日志
- 记录使用日志
- 商品退回
- 按日使用次数限制
- 阶梯价格、库存和稀有度等规则
- 类抽卡玩法

依赖状态：

- `treasures`
- `storage`
- `logs`
- `redemptionHistory`
- `dailyRedemptionCounts`
- `dailyUsageCounts`
- `students`
- `history`

迁移判断：

- 属于 P1
- 模块商业化潜力高，适合后续产品包装
- 当前库存、订单、使用、退款耦合在前端，必须重构成正式商城/库存域模型

### 4.6 学生头像 / 档案

入口：

- `Nav.profile`
- `ProfileView`

已实现能力：

- 学生头像上传
- 学生称号设置
- 查看学生历史与部分档案信息

依赖状态：

- `students`
- `history`

迁移判断：

- 属于 P0
- 新系统中应升级为“学生档案”模块，不应只保留头像页的概念

### 4.7 设置 / 维护中心

入口：

- `Nav.settings`
- `SettingsView`

已实现能力：

- 系统维护操作集中入口
- 生成系统简报
- 导入导出学生 Excel
- 导入导出积分 Excel
- 导入导出藏宝阁 Excel
- 导入导出考勤 Excel
- 导入导出语录 Excel
- 导入导出系统配置
- JSON 导入导出
- 手动快照
- 快照恢复
- 从历史恢复分数
- 手工修分
- 系统重置
- 测试模式开关
- 桌面端相关探测和保存

依赖状态：

- `students`
- `history`
- `config`
- `attendanceRecords`
- `treasures`
- `storage`
- `logs`
- `quotes`
- `messages`
- `teacherMessages`
- `redemptionHistory`
- `dailyRedemptionCounts`
- `dailyUsageCounts`
- `tasks`
- `battle`

迁移判断：

- 其中一部分属于 P0，一部分属于 P1/P2
- 当前页面承担了过多运维职责，后续应拆为：
  - 系统配置
  - 数据导入导出
  - 数据修复与管理员工具

### 4.8 双子星对战

- `Nav.battle`

依赖状态：

- `battle`
- `students`

- 是有特色的玩法模块，但不是基础可用性的阻塞项
- 适合在核心模块稳定后再迁移

## 5. 系统级基础能力

### 5.1 本地缓存

已实现能力：

- `class_manager_data`
- `attendance_records`
- `class_treasure_data`
- `class_manager_snapshots`
- `cm_offline_snapshot`

用途：

- 提升刷新后恢复能力
- 本地离线保存
- 快照和历史恢复

迁移判断：

- 属于必须重构，不应原样继承
- 新系统应改为更小粒度缓存和操作队列

### 5.2 自动保存

已实现能力：

- 页面状态变更后 1.5 秒防抖保存
- 同时写本地和远端

迁移判断：

- 核心价值需要保留
- 实现方式需要改为按实体写入，而不是整包状态保存

### 5.3 手动刷新与自动刷新

已实现能力：

- 手动从服务器拉取数据
- 自动刷新
- 防止脏数据覆盖的 `isSavingRef` / `isDirtyRef` 逻辑

迁移判断：

- 价值保留
- 实现需改为基于版本、更新时间和实体查询

### 5.4 离线快照与冲突处理

已实现能力：

- 断网时写离线快照
- 重连后比较本地与服务器时间
- 允许用户选择：
  - 本地覆盖服务器
  - 服务器覆盖本地
  - 合并后同步

迁移判断：

- 能力有价值，但当前实现风险高
- 新系统建议改为“操作日志同步”而不是“大对象覆盖合并”

### 5.5 自动快照

已实现能力：

- 自动生成快照
- 手动生成快照
- 保留最近 10 份快照

迁移判断：

- 应保留
- 新系统应把“快照”和“数据库备份”明确区分

### 5.6 测试模式

已实现能力：

- 模拟时间
- 调整时间流速
- 使用内存态替代本地真实存储

迁移判断：

- 属于 P2
- 对开发测试有帮助，但不属于第一期产品能力

### 5.7 自动惩罚衰减

已实现能力：

- 按周自动衰减学生 penalty
- 支持冻结态控制

迁移判断：

- 属于业务规则增强
- 应迁移到后端策略服务，不应继续依赖前端定时触发

- 商品配置

- 属于 P0

### 6.2 语录与消息

已实现能力：

- 默认励志语录
- 教师消息

- 属于 P1

## 8. 功能迁移优先级建议

### 第一批迁移

- 认证与用户
- 班级/学生档案
- 积分系统
- 系统配置
- 考勤

原因：

- 这是系统可用性的最小闭环

### 第二批迁移

- 作业登记
- 任务系统
- 商城/藏宝阁
- 管理后台
- 快照与导入导出

原因：

- 这些功能重要，但可以在最小闭环后补齐

### 第三批迁移

- 双子星对战
- 测试模式
- 桌面端相关辅助能力

原因：

- 非基础阻塞能力
- 适合在底层稳定后重做

## 9. 建议废弃或重构而非直接保留的实现

以下实现不建议原样继承：

- 前端单文件业务集中实现
- 基于 `window` 的全局工具函数
- 本地存储中长期保存整套业务主数据
- 全量 JSON 持久化到后端
- 前端直接修改核心业务状态作为事实来源
- 用页面逻辑承担审计、冲突解决和数据恢复的主要责任

## 10. 本文档结论

### 10.1 必须保留的业务价值

- 多用户班级管理
- 学生档案
- 积分流水
- 考勤管理
- 可配置规则
- 管理员能力
- 基础数据导入导出

### 10.2 需要重构保留的增强能力

- 任务系统
- 商城/藏宝阁
- 快照恢复
- 留言/公告
- 对战系统
