# 系统重构蓝图 v1

## 1. 文档目的

本文档用于为 `classmanager-multi` 的重构与重写提供统一基线。目标不是继续在现有实现上做局部修补，而是在保留业务价值与历史数据的前提下，重建系统底层架构，使其满足以下要求：

- 更快的页面加载速度
- 更稳定的运行与更低的数据损坏风险
- 更清晰的业务边界与更低的维护成本
- 更强的安全性、可审计性和扩展性
- 为未来商业化和多租户 SaaS 形态预留空间

本文档面向后续的产品设计、技术设计、数据库设计、迁移开发和测试验收。

## 2. 对现有系统的判断

### 2.1 当前系统的真实形态

现有系统本质上是一个“前端承载绝大多数业务逻辑，后端仅承担认证与 JSON 存储”的单体工具系统：

- 前端为单页面静态应用，核心业务集中在单文件脚本中
- 后端为 Express 服务
- 存储为 SQLite
- 认证为 JWT
- 多用户能力已具备，但业务数据仍以大对象快照方式保存

### 2.2 当前系统的主要问题

1. 业务边界错位  
   业务规则主要在前端，后端无法真正约束数据合法性。

2. 数据模型不稳定  
   大量核心状态以 JSON 大对象存储，字段隐式增长，缺少模式约束。

3. 单文件复杂度过高  
   代码耦合严重，修改一个功能时难以判断副作用范围。

4. 同步机制风险高  
   当前以整包数据读写、离线覆盖、自动刷新为主，冲突合并复杂且脆弱。

5. 安全能力薄弱  
   缺少租户隔离、权限模型、审计日志、密钥治理和默认弱口令治理。

6. 不具备商业化底座  
   当前结构更接近个人工具，不适合直接演化为 SaaS 产品。

### 2.3 重构原则

- 不在旧系统上继续大规模叠加功能
- 重构优先重建边界和数据模型，而不是先做 UI 美化
- 先稳定领域模型，再实现功能迁移
- 先支持双轨迁移，再考虑全面替换
- 所有关键业务必须具备审计、校验、事务和测试

## 3. 重构目标

### 3.1 业务目标

- 维持并升级当前班级管理核心能力
- 保留已有特色功能，例如积分、考勤、任务、藏宝阁、对战
- 让每个模块可独立维护、独立扩展、独立测试

### 3.2 技术目标

- 使用结构化数据库替代 JSON 主存储
- 使用模块化前后端架构替代单文件富前端
- 将核心业务规则迁移到后端领域服务
- 将同步模型从“全量覆盖”改为“按实体更新”

### 3.3 商业化目标

- 支持多租户
- 支持角色权限
- 支持配置开关与套餐能力
- 支持报表与审计
- 支持后续的订阅、支付和私有化部署

## 4. 目标架构

### 4.1 推荐技术选型

#### 方案 A：更适合长期商业化

- 前端：Next.js + TypeScript
- 后端：NestJS + TypeScript
- 数据库：PostgreSQL
- ORM：Prisma
- 缓存/队列：Redis
- 对象存储：S3 兼容服务
- API 文档：OpenAPI
- 部署：Docker + CI/CD

#### 方案 B：更轻量但仍可维护

- 前端：React + Vite + TypeScript
- 后端：Fastify + TypeScript
- 数据库：PostgreSQL
- ORM：Prisma

本项目推荐采用方案 A。原因如下：

- 更适合构建清晰模块与依赖边界
- 更适合多人协作与长期维护
- 更适合后续权限、审计、多租户、异步任务的扩展

### 4.2 逻辑架构

新系统应拆分为以下层次：

1. 表现层  
   Web 页面、表单、图表、管理界面、移动端适配。

2. 应用层  
   负责编排业务流程，例如“登记考勤”“批量加分”“兑换商品”。

3. 领域层  
   负责领域实体、规则、校验、状态转移和领域事件。

4. 基础设施层  
   负责数据库、缓存、文件存储、消息队列、邮件、短信、日志等。

5. 运维治理层  
   负责监控、备份、告警、权限审计、风控、租户管理。

### 4.3 系统边界

新系统至少分为以下逻辑子系统：

- 认证与租户中心
- 班级与成员管理
- 积分与流水中心
- 考勤中心
- 作业与任务中心
- 商城与库存中心
- 对战与活动中心
- 配置与功能开关中心
- 报表与审计中心
- 数据导入导出与迁移中心

## 6. 数据模型设计原则

### 6.1 核心原则

- 当前状态与历史流水分离
- 所有关键写操作必须保留操作来源
- 所有关键业务实体有明确主键、版本号和时间戳
- 重要业务支持软删除或归档
- 配置与业务数据分离
- 大对象 JSON 仅可作为补充字段，不可作为主事实来源

### 6.2 建议的核心数据表

#### 组织与权限

- `tenants`
- `users`
- `memberships`
- `roles`
- `permissions`
- `role_permissions`

#### 班级与学生

- `classes`
- `students`
- `student_groups`
- `student_dormitories`
- `student_positions`

#### 积分系统

- `point_accounts`
- `point_transactions`
- `point_reason_templates`
- `point_rules`
- `point_batches`

#### 考勤系统

- `attendance_schedules`
- `attendance_sessions`
- `attendance_records`
- `attendance_policies`

#### 作业与任务

- `subjects`
- `homework_tasks`
- `homework_submissions`
- `tasks`
- `task_claims`

#### 商城与库存

- `shop_items`
- `inventory_balances`
- `redemption_orders`
- `item_usage_records`

#### 对战系统

- `battle_seasons`
- `battle_teams`
- `battle_matches`
- `battle_settlements`
- `battle_rule_sets`

#### 平台治理

- `feature_flags`
- `audit_logs`
- `snapshots`
- `import_jobs`
- `export_jobs`

### 6.3 关键建模约束

1. 积分不可直接手改余额  
   余额由流水聚合或事务更新得出。

2. 考勤记录与积分结果分开  
   原始考勤是事实，积分扣减是派生结果。

3. 商品库存与兑换记录分开  
   库存只表示当前量，兑换与使用保留为明细。

4. 对战规则与赛季数据分开  
   同一玩法未来可配置不同规则版本。

5. 学生组织关系使用独立关联表  
   避免学生实体承担过多耦合字段。

## 7. API 设计原则

### 7.1 风格

- 优先 REST
- 局部复杂查询可考虑 GraphQL 或聚合查询接口，但不作为第一期主路线

### 7.2 约束

- 所有输入使用 schema 校验
- 所有写操作使用事务
- 所有重要写操作保留 `actor_id`
- 幂等写接口支持 `idempotency_key`
- 所有分页查询统一规范
- 所有错误返回统一结构

### 7.3 示例 API 边界

- `/auth/*`
- `/tenants/*`
- `/classes/*`
- `/students/*`
- `/points/*`
- `/attendance/*`
- `/homework/*`
- `/tasks/*`
- `/shop/*`
- `/battle/*`
- `/reports/*`
- `/admin/*`

## 8. 前端重构方案

### 8.1 前端目标

- 页面拆分按业务模块组织
- 服务端状态与本地交互状态分离
- 避免“单文件 + 巨量 useState”
- 明确表单、查询、列表、图表和管理台边界

### 8.2 推荐目录结构

```text
apps/web
  src/
    app/
    modules/
      auth/
      classes/
      students/
      points/
      attendance/
      homework/
      tasks/
      shop/
      battle/
      settings/
      reports/
    components/
      ui/
      layout/
    lib/
      api/
      auth/
      schema/
      utils/
```

### 8.3 前端状态管理原则

- 服务端状态：TanStack Query
- 表单状态：React Hook Form + Zod
- 页面局部状态：Zustand 或组件内状态
- 不再把整套业务主数据长期保存在 localStorage

### 8.4 离线策略

保留“离线增强”，放弃“离线即主存储”的模式：

- 只缓存最近查询结果和待同步操作队列
- 本地保存的是“操作日志”，不是全量业务快照
- 冲突解决基于实体版本和时间戳
- 不允许整包覆盖远端主数据

## 9. 后端重构方案

### 9.1 模块化方式

后端按领域模块组织：

- auth
- tenant
- class
- student
- point
- attendance
- homework
- task
- shop
- battle
- report
- audit
- config

### 9.2 关键工程约束

- 所有模块 DTO 和 schema 明确
- 所有 service 可单元测试
- 所有 repository 仅负责持久化
- 领域逻辑不写在 controller
- 所有数据库变更通过 migration 管理

### 9.3 事务边界

以下操作必须事务化：

- 批量加减分
- 考勤登记并派生积分
- 商品兑换与库存扣减
- 任务认领与奖励发放
- 对战结算
- 数据迁移导入

## 10. 安全设计

### 10.1 认证与授权

- 禁止默认管理员弱口令
- 强制通过环境变量提供密钥
- Access Token + Refresh Token
- 密码加盐哈希
- 支持角色权限矩阵
- 支持租户级访问控制

### 10.2 数据安全

- PostgreSQL 主库
- 自动备份
- 跨版本迁移脚本
- 敏感操作保留审计
- 重要记录避免物理删除

### 10.3 运维安全

- HTTPS
- 访问日志
- 错误监控
- 接口限流
- 登录失败风控

## 11. 性能设计

### 11.1 前端性能

- 路由级拆包
- 首屏按需加载
- 大列表分页或虚拟滚动
- 图表延迟加载
- 配置与字典缓存

### 11.2 后端性能

- 建立合理索引
- 统计接口做预聚合
- 热点配置走缓存
- 大导出任务异步化

### 11.3 数据访问优化

- 不再全量读取整套班级状态
- 每个页面只拉取必要资源
- 支持分页、筛选、排序
- 使用版本号与更新时间做增量同步

## 12. 可商业化设计

### 12.1 多租户模型

系统要从第一版开始具备租户概念：

- 一个租户可对应一个班级、学校或机构
- 数据按租户隔离
- 用户可加入多个租户
- 功能开关与套餐按租户控制

### 12.2 商业化版本形态

- 个人班主任版
- 学校版
- 机构版
- SaaS 订阅版
- 私有化部署版

### 12.3 为商业化预留的能力

- 套餐与计费
- 超级管理员后台
- 租户生命周期管理
- 操作审计
- 数据导出
- 白标与定制配置

## 13. 重构实施策略

### 13.1 总策略

采用“双轨迁移”：

- 旧系统继续可运行，但冻结大功能新增
- 新系统独立开发
- 按模块迁移数据和业务
- 完成灰度验证后切换

### 13.2 分期计划

#### 第 0 期：冻结与治理

目标：

- 冻结旧系统的大功能扩展
- 建立数据备份机制
- 修复高危安全问题
- 梳理数据字典与业务词典

输出：

- 现有功能清单
- 现有字段字典
- 旧数据导出脚本
- 风险清单

#### 第 1 期：新底座搭建

目标：

- 搭建新前后端仓库
- 建立 PostgreSQL schema
- 建立鉴权、租户、班级、学生、积分核心模块
- 建立审计、日志、配置、迁移框架

输出：

- 新仓库基础工程
- 基础表结构
- 核心模块 API
- CI/CD 和测试框架

#### 第 2 期：核心模块迁移

建议顺序：

1. 认证与权限
2. 班级与学生
3. 积分流水
4. 考勤
5. 作业与任务
6. 商城/藏宝阁
7. 设置中心
8. 对战系统

每个模块都应包括：

- 新数据模型
- 新 API
- 新前端页面
- 旧数据迁移脚本
- 回归测试

#### 第 3 期：灰度切换

目标：

- 选择一个真实班级或测试租户试运行
- 对关键数据进行双写或比对
- 修复迁移偏差

#### 第 4 期：正式替换

目标：

- 旧系统切换为只读或归档模式
- 新系统成为主入口
- 保留旧导出能力一段过渡期

## 14. 数据迁移方案

### 14.1 迁移原则

- 迁移业务意义，不机械复制旧结构
- 先导出旧数据，再映射，再导入新表
- 每类数据迁移要可重复执行
- 所有迁移过程必须可审计、可回放

### 14.2 旧数据到新模型的映射思路

#### 用户与租户

- 旧 `users` 映射到新 `users`
- 每个现有班级用户可先映射为一个独立租户或独立班级管理员

#### 学生数据

- 旧 `students` JSON 拆分为 `students`、`student_groups`、`student_dormitories`、`student_positions`

#### 积分历史

- 旧 `history` 映射到 `point_transactions`
- 原 `snapshot` 字段只作为迁移辅助，不作为新系统事实字段

#### 考勤

- 旧 `attendanceRecords` 拆分为 `attendance_sessions` 和 `attendance_records`
- 自动派生的加减分单独生成流水

#### 宝物与库存

- 旧 `treasures` 映射到 `shop_items`
- 旧 `storage` 映射到 `inventory_balances`
- 旧 `logs` 和 `redemptionHistory` 映射到 `redemption_orders`、`item_usage_records`

#### 对战

- 旧 `battle` 拆分到赛季、队伍、对局、结算表

### 14.3 迁移执行流程

1. 从旧库导出用户级完整 JSON
2. 进行字段清洗与模式校验
3. 生成中间迁移文件
4. 将中间文件导入 PostgreSQL
5. 运行迁移校验脚本
6. 生成人工抽检报告

## 15. 测试与质量保障

### 15.1 必备测试类型

- 单元测试
- 集成测试
- 迁移测试
- 回归测试
- 权限测试
- 并发和事务测试

### 15.2 高风险场景测试

- 批量加分/扣分
- 考勤登记重复提交
- 商品库存并发兑换
- 对战结算重复执行
- 网络失败后的重试
- 离线后恢复同步
- 权限越权访问

### 15.3 上线门槛

以下条件必须满足后才能切换主系统：

- 核心迁移脚本可重复执行
- 关键模块测试覆盖达标
- 至少一轮真实数据演练完成
- 审计日志和备份机制可用
- 灰度用户试运行通过

6. 建立新系统空仓和基础工程

## 18. 本文档的决策结论

### 18.1 已确定

- 不继续以现有单文件前端为长期演进基础
- 不继续以 SQLite + JSON 大对象作为商业化主存储
- 新系统采用模块化前后端架构
- 新系统以 PostgreSQL + 结构化模型为核心
- 旧系统采用双轨迁移而不是一次性替换

### 18.2 待后续细化

- 租户模型的最小商业版本
- 权限矩阵详细设计
- 对战模块的最终保留范围
- 离线能力的精确交互方案
- 报表与导出优先级

## 19. 附录：旧系统到新系统的总迁移顺序

推荐顺序如下：

1. 用户与认证
2. 租户与班级
3. 学生档案
4. 积分流水
5. 考勤
6. 作业
7. 任务
8. 商城
9. 设置
10. 对战
11. 审计与报表

该顺序的原因是：

- 先建立主身份和组织边界
- 再迁移最核心、最频繁使用的数据
- 最后处理玩法型和扩展型模块
