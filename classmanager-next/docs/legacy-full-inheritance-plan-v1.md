# 旧系统功能完整继承推进计划 v1

## 1. 文档目的

本文档用于把“旧系统功能是否已被新系统完整继承”的审计结论，整理成一份可执行的后续推进文档，避免后续只凭口头结论继续开发。

本文档主要回答三个问题：

- 当前到底能不能说“新系统已经完整替代旧系统”
- 还差哪些缺口
- 应该按什么顺序继续做

配套文档：

- 功能对账基线：`docs/legacy-feature-progress-matrix-v1.md`
- 阶段性人工回归：`docs/upgrade-regression-checklist-v1.md`
- 兼容区数据清单：`docs/legacy-compat-data-inventory-v1.md`

## 2. 当前判断

截至当前，新系统**不能直接判定为“已完整等价实现旧系统全部功能”**。

更准确的判断是：

- 核心教学管理功能已经基本可用，并能承接日常主要场景
- 旧系统的任务、藏宝阁 / 祈愿、双子星虽然已在新系统可用，但仍主要处于“兼容承接态”
- 部分高频写操作、迁移恢复治理、后台治理与权限治理仍需继续收口

按当前状态，可分成两个层级理解：

### 2.1 已接近日常可替代

以下能力已经基本可以支持日常班级使用：

- 登录认证
- 学生档案
- 积分总览与基础写操作
- 考勤总览、统计与大部分高频写操作
- 作业总览、统计与基础登记写操作
- 设置总览与大部分维护工具
- 管理后台基础能力
- 导出、恢复、迁移主链路

### 2.2 仍不能算完整等价继承

以下能力仍不能视为“正式原生模块已完全替代旧系统”：

- 任务系统
- 藏宝阁 / 商城 / 祈愿
- 对战系统 / 双子星
- 更复杂批量写操作与边界治理
- 更完整的二次迁移 / 灾备恢复演练
- 更细的后台治理与权限收口

## 3. 推进原则

后续推进顺序不应以“先重构得漂亮”为优先，而应以“先能稳定替代旧系统”为优先。

建议遵循以下原则：

1. 先补会直接影响旧系统替代的高频缺口
2. 再补迁移、恢复、二次迁移闭环
3. 最后再把兼容区逐步拆成正式领域模块

换句话说：

- 先解决“能不能完全接住旧业务”
- 再解决“能不能稳定迁移和恢复”
- 最后解决“是不是已经完全去旧系统化”

## 4. 缺口分级

## 4.1 P0：影响日常替代的缺口

这部分不补齐，就很难说新系统已经可以稳定替代旧系统日常使用。

### P0-1 旧功能兼容区仍未正式化

当前最明显的缺口是以下模块仍主要挂在兼容区：

- 任务
- 藏宝阁 / 祈愿
- 双子星

当前目标不一定是立刻重做成全新模块，但至少要做到：

- 旧流程可以稳定跑通
- 关键边界场景有自动回归
- 前后端提示、权限、冻结态、异常处理一致
- 能明确区分哪些是“已可用”，哪些是“仅兼容承接”

### P0-2 核心模块剩余高频写操作仍需收口

虽然积分、考勤、作业都已具备主要写链路，但仍未到“旧系统能力全部闭环”的程度：

- 积分：更复杂批量加减分、历史批次治理、撤销边界
- 考勤：更复杂规则、状态治理、跨场次修正边界
- 作业：更完整的批量修正、批量撤销与历史治理

这部分是最接近日常真实使用的缺口，应作为兼容区之外的第一优先级。

## 4.2 P1：影响迁移恢复可信度的缺口

这部分不一定阻塞日常使用，但会影响“旧数据完整迁移 / 支持二次迁移 / 支持恢复演练”的可信度。

主要包括：

- 旧维护配置 JSON 的逐项兼容回归仍需继续补齐
- `structured full export / restore` 仍需继续做覆盖恢复与重复恢复演练
- 二次迁移幂等性仍需继续做真实数据回归
- 导出文件治理、灾备边界与更大范围恢复能力仍需继续收口

这部分的目标不是再补一个“能跑通一次”的脚本，而是做到：

- 可重复执行
- 能发现脏 mapping
- 出错后可恢复
- 多次导入 / 恢复后数据仍稳定

## 4.3 P2：影响正式运维托管的缺口

这部分主要影响后期的治理能力，而不是最早的业务承接能力。

主要包括：

- 后台成员治理继续收口
- 权限细化与跨模块一致性
- 运维工具继续补齐
- 更完整的审计与验收支撑

这部分应排在 P0、P1 之后，但正式切换前仍应完成。

## 5. 建议开发顺序

## 5.1 第 1 轮：先把兼容区做成“稳定可替代”

目标：

- 先保证旧系统最独特的兼容玩法不掉链子
- 先做到“旧流程能稳定接住”，再考虑是否重构成正式模块

建议任务：

- 补 `legacy` 模块全链路自动回归：
  - 任务领取
  - 藏宝阁兑换
  - 藏宝阁使用 / 退回
  - 祈愿
  - 双子星初始化 / 备份导入导出 / 考试导入 / 结算
- 补冻结态、无权限、重复操作、库存 / 余额 / 日限等边界回归
- 整理兼容区当前仍依赖 `legacyCompat` 直接维护的数据清单

完成标志：

- 旧功能兼容区的核心主流程都有自动测试覆盖
- 不再依赖人工记忆判断兼容功能是否“还能用”

## 5.2 第 2 轮：补核心教学模块剩余高频写操作

目标：

- 让学生、积分、考勤、作业真正达到高频场景可平替

建议任务：

- 积分：
  - 补复杂批量场景与历史批次治理
  - 继续收口撤销边界
- 考勤：
  - 补更复杂规则和状态回归
  - 继续收口跨场次修正 / 缺勤结算边界
- 作业：
  - 补更完整的批量修正与批量撤销治理

完成标志：

- 核心教学管理高频写场景不再明显弱于旧系统

## 5.3 第 3 轮：补迁移、恢复、二次迁移闭环

目标：

- 让“旧数据完整迁移”和“出问题后可恢复”都具备可信度

建议任务：

- 对旧维护配置导入做字段级回归清单
- 做 `structured full export / restore` 覆盖恢复演练
- 做二次迁移幂等回归：
  - 小组 / 宿舍 / 岗位
  - 考勤 phase2
  - 积分流水
  - 头像 mapping
  - 其他 `migrationMapping` 敏感域

完成标志：

- 同一批旧数据可多次迁移
- 恢复后仍可继续迁移
- mapping 异常和脏数据能被识别并收口

## 5.4 第 4 轮：收口后台治理和权限

目标：

- 让系统从“能用”进入“可托管、可运维”

建议任务：

- 把成员、角色、操作能力整理成更清晰的权限矩阵
- 收口冻结态、导入导出、恢复、兼容区写操作的权限一致性
- 补后台运维视角的回归清单

完成标志：

- 管理后台能支持真实班级切换后的日常治理

## 5.5 第 5 轮：最后再去兼容化

目标：

- 把兼容区逐步从“业务主入口”降级为“迁移兼容区”

建议任务：

- 逐步把任务、藏宝阁、双子星拆成正式领域模型
- 逐步减少 `legacyCompat` 承载关键业务写操作
- 把兼容区收缩到只读展示、迁移承接或少量维护入口

完成标志：

- 旧系统功能不再主要依赖兼容区运行
- 新系统可以更接近“完整原生替代旧系统”

## 6. 建议验证基线

每轮开发完成后，至少执行以下验证：

```bash
npm run build -w @classmanager/web
npm test -w @classmanager/api
```

如果本轮涉及专项模块，建议再补充对应定向检查：

- `legacy` 兼容功能专项回归
- `points` / `attendance` / `homework` / `settings` / `admin` 模块专项回归
- 人工走查 `docs/upgrade-regression-checklist-v1.md`

## 7. 阶段完成标志

## 7.1 可宣称“基本可替代旧系统日常使用”

满足以下条件时，可以较有把握地说新系统已基本可替代旧系统日常使用：

- 兼容区核心玩法稳定可用
- 学生、积分、考勤、作业、设置主链路高频写场景已收口
- 构建与 API 测试稳定通过
- 人工回归清单中的高频入口均可正常进入

## 7.2 可宣称“已完整继承旧系统功能”

满足以下条件时，才更接近宣称新系统已完整继承旧系统功能：

- 兼容区不再承载关键业务主流程
- 旧系统核心玩法已有正式领域建模或稳定承接方案
- 二次迁移、恢复、导出、灾备均经过重复演练
- 后台治理、权限治理、审计治理均已收口

## 8. 当前建议

按当前状态，最合理的推进顺序是：

1. 先补 `legacy` 兼容区回归和边界治理
2. 再补积分 / 考勤 / 作业剩余高频写操作
3. 再补迁移恢复与二次迁移演练
4. 再收口后台治理与权限
5. 最后再逐步去兼容化

一句话总结：

**当前新系统已经接近“核心日常可替代”，但还不能算“旧系统全部功能已完整原生继承”；后续应优先补替代能力，再补恢复能力，最后再去兼容化。**
