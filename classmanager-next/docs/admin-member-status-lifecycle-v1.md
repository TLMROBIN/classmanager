# 后台成员状态生命周期说明 v1

## 1. 当前目的

当前后台已经能读到成员状态，也能在摘要中看到：

- `active`
- `disabled`
- `invited`

当前系统已经开放最小写操作：

- `active -> disabled`
- `disabled -> active`

但还没有开放 `invited` 相关写操作。进入下一阶段前，仍然需要把状态语义和限制固定，避免后面把“禁用成员”“移除成员”“邀请成员”混成一类。

## 2. 三种状态的定义

### `invited`

表示租户内已经预留成员关系，但用户尚未完成首次登录或激活。

当前阶段约束：

- 可以出现在成员列表和后台摘要中
- 不应拥有完整业务写权限
- 不应被计入可正常工作的活跃成员数

### `active`

表示当前成员已可正常登录并使用租户内授权能力。

当前阶段约束：

- 可参与权限判定
- 可出现在业务审计中
- 可被后台角色分配逻辑读取和修改角色

### `disabled`

表示成员关系保留，但不应继续拥有正常使用权限。

当前阶段约束：

- 应保留历史审计和角色变更记录
- 登录后不应再获得租户访问能力
- 不应被当作可正常工作的成员统计

## 3. 后续写操作建议顺序

不要一次性开放“邀请、停用、恢复、删除”。建议按下面顺序推进：

1. 先做 `active -> disabled`
2. 再做 `disabled -> active`
3. 再做 `invited -> active` 的激活完成链路
4. 最后才考虑成员删除或彻底移出租户

原因：

- `disabled` 是最接近旧系统后台管理的低风险能力
- 恢复链路可以复用停用的保护规则
- `invited` 往往会连到账号创建、邮件/短信、首次登录等更复杂流程
- 删除最危险，应该最后做

## 4. 建议保护规则

当前已实现或应持续保持的保护：

- 禁止停用本人当前成员关系
- 禁止停用最后一个 `tenant_owner`
- 当前阶段不支持修改 `invited` 成员状态
- 禁止直接删除仍有历史审计的成员
- 状态变更必须写独立审计动作，例如：
  - `membership.status.disable`
  - `membership.status.enable`
  - `membership.invite.create`

## 5. 前端展示建议

当前前端只读页已经具备这些基础：

- 成员状态筛选
- 后台摘要中的状态计数
- 成员详情中的当前状态

当前前端已经提供最小交互：

- 在成员详情页增加“停用成员 / 恢复成员”
- 对 `invited` 成员先只读展示，不立即开放复杂邀请流程
- 所有状态写操作都必须二次确认

## 6. 当前结论

成员状态管理已经具备进入实现阶段前的最小前置条件：

- 后端已有状态数据
- 前端已有状态可见性
- 后台摘要已有状态计数

下一步若继续做后台写操作，优先项应是：

1. `invited` 成员的创建/邀请准备结构
2. 成员状态写操作的真实数据库回归
3. 对应审计与权限保护继续补强
