# 导入导出架构方案 v1

## 1. 文档目的

本文档用于替代旧系统中继承自单用户单文件时代的“大 JSON 打包导出”模式。

目标：

- 降低导出导入导致的数据损坏风险
- 区分业务导出、迁移导出和运维备份
- 为后续审计、版本控制和异步任务提供结构约束

## 2. 为什么不能沿用旧方案

旧系统的导出模式本质上是：

- 把多个业务域状态拼成一个大对象
- 一次性导出到单个 JSON / Excel
- 导入时按前端字段结构直接覆盖

这种模式的问题：

- 字段强依赖前端内部结构
- 缺乏版本声明
- 无法按领域恢复
- 无法做细粒度校验
- 容易把局部错误放大成整包损坏

## 3. 新架构的三类导出

### 3.1 业务导出

面向老师和管理者。

输出内容：

- 学生档案
- 积分流水
- 考勤记录
- 作业登记
- 班级配置清单

特点：

- 人可读
- 可下载为 Excel / CSV / PDF
- 只表达业务结果，不承载系统恢复职责

### 3.2 迁移导出

面向旧系统迁移和回滚。

输出内容：

- 结构化 JSON
- 按领域拆包
- 带 schema 版本
- 带导出时间和来源系统信息

特点：

- 机器可读
- 可校验
- 可 dry-run
- 可重复导入

### 3.3 运维备份

面向灾备与恢复。

输出内容：

- PostgreSQL 逻辑备份
- 对象存储文件备份
- 配置快照
- 审计日志快照

特点：

- 不给普通业务用户使用
- 由系统定时执行
- 用于灾难恢复，不代替业务导出

## 4. 导出包设计原则

### 4.1 每个导出包必须带元信息

至少包含：

- `exportType`
- `schemaVersion`
- `tenantId`
- `classId`
- `exportedAt`
- `sourceSystem`

### 4.2 不再导出“全量前端状态”

禁止继续导出：

- `students + history + config + attendanceRecords + ...` 这种混合大对象

改为：

- `students.json`
- `point-transactions.json`
- `attendance-sessions.json`
- `attendance-records.json`
- `settings.json`

### 4.3 导入前必须先校验

导入流程固定为：

1. 上传导入包
2. 校验 schema 版本
3. 校验字段合法性
4. 生成 dry-run 结果
5. 用户确认
6. 事务执行导入
7. 生成导入报告与审计日志

## 5. 迁移导入设计原则

### 5.1 分领域导入

建议顺序：

1. 组织与学生
2. 积分流水
3. 考勤
4. 作业
5. 其他模块

### 5.2 幂等

每次导入必须可追踪和防重：

- 导入任务表
- 外部主键映射表
- 领域级唯一约束

### 5.3 显式确认

真实写入必须：

- 不是默认行为
- 必须带确认参数
- 必须落审计日志

## 6. 业务导出建议格式

### 6.1 优先 Excel / CSV 的场景

- 积分流水
- 考勤明细
- 作业登记
- 学生名单

### 6.2 优先 JSON 的场景

- 迁移导出
- 配置导出
- 模块规则导出

## 7. 配置导入导出建议

配置同样分层：

- 系统配置导出
- 模块配置导出

禁止：

- 把全部配置和业务数据混在同一个导入包中

## 8. 审计要求

所有导入导出任务都应记录：

- 谁触发
- 导出/导入什么内容
- 数据范围
- 文件名
- 开始时间
- 结束时间
- 结果摘要
- 错误信息

## 9. 当前实施建议

### 9.1 短期

- 保持现有迁移脚本的 `export -> transform -> validate -> dry-run -> apply` 链路
- 新系统的业务导出先从只读 API + Excel/CSV 开始

### 9.2 中期

- 增加统一导入导出任务中心
- 导出改成异步任务
- 导入报告写入数据库

### 9.3 长期

- 接入对象存储
- 接入定时备份
- 增加租户级恢复演练

## 10. 当前明确不做的事

- 不再恢复旧系统那种“一键整包覆盖导入”
- 不把前端内部状态作为导入标准
- 不让业务用户直接接触数据库备份文件
