# 学生写操作实现说明 v1

## 1. 当前范围

当前只补了一条切换价值高、范围可控的学生写能力：

- 新增学生
- 学生基础资料编辑
- 学生主小组 / 主宿舍 / 岗位调整
- 独立学生状态字典接入
- 学生安全删除

这一版故意不做：

- 头像与档案图片

## 2. 后端接口

文件：

- `apps/api/src/modules/classes/routes.ts`

当前新增接口：

- `POST /api/classes/:classId/students`
- `PUT /api/students/:studentId`
- `DELETE /api/students/:studentId`
- `PUT /api/students/:studentId/organization`

新增学生规则：

- 班级必须存在
- 必须先通过班级访问校验
- 当前复用积分管理权限
- 班级冻结时拒绝写入
- 同班级下 `studentNo` 必须唯一
- 如果未传 `sortOrder`，则自动追加到当前末尾
- 创建学生时同步创建空积分账户
- 同步写 `student.create` 审计

当前允许修改：

- `studentNo`
- `name`
- `gender`
- `status`
- `sortOrder`

组织归属规则：

- 当前只允许修改：
  - 主小组
  - 主宿舍
- 岗位
- 允许清空当前主小组或主宿舍
- 允许清空当前岗位集合
- 目标小组和宿舍必须属于当前班级且仍为启用状态
- 目标岗位必须属于当前班级且仍为启用状态
- 班级冻结时拒绝写入
- 内容未变化时拒绝提交
- 同步写 `student.organization.update` 审计

写入规则：

- 学生必须存在
- 必须先通过班级访问校验
- 当前复用积分管理权限
- 班级冻结时拒绝写入
- 同班级下 `studentNo` 必须唯一
- 内容未变化时拒绝提交
- 同步写 `student.update` 审计

删除规则：

- 学生必须存在
- 必须先通过班级访问校验
- 当前复用积分管理权限
- 班级冻结时拒绝删除
- 若已有积分流水（含作业 / 考勤衍生记录），拒绝物理删除
- 若已有考勤记录，拒绝物理删除
- 若积分账户仍有非零分值，拒绝物理删除
- 删除前会自动清理班级配置中的现代引用：
  - 值日安排
  - 科目代表
  - 心理委员名单
  - 学生会岗位绑定
- 删除前也会自动清理低风险兼容运行态引用：
  - 任务领取状态
  - 藏宝阁仓库条目
  - 藏宝阁兑换历史条目
  - 双子星战队成员
- 若清理上述现代引用后，班级配置或兼容区仍残留学生 ID 引用，则拒绝删除
- 兼容历史归档类引用（例如双子星考试快照）当前仍会继续阻塞物理删除
- 删除时会同步清理 `student` / `student_avatar` 迁移映射
- 同步写 `student.delete` 审计

## 3. 前端能力

文件：

- `apps/web/src/components/StudentDetailPanel.tsx`
- `apps/web/src/App.tsx`
- `apps/web/src/lib/api.ts`

当前页面能力：

- 在学生列表页直接新增学生
- 在学生列表页按姓名 / 学号 / 小组 / 宿舍 / 状态筛选学生
- 学生列表支持当前筛选结果按状态字典动态汇总摘要
- 学生列表新增与批量状态调整已改为读取设置页学生状态字典，支持自定义显示名称与参与日常标记
- 学生列表支持批量状态调整
- 在学生详情页直接编辑基础资料
- 在学生详情页直接调整主小组、主宿舍和岗位
- 学生详情顶部会展示当前状态标签，基础资料编辑也复用状态字典选项，并提示是否默认参与日常生成
- 组织归属调整区支持当前值 / 保存后预览
- 学生详情新增删除治理区，实时展示“可删 / 不可删”阻塞原因
- 若允许删除，前端会在确认框中提示即将自动清理的配置引用
- 学生详情最近积分记录支持按类型、撤销状态、日期范围和关键词筛选
- 学生详情最近积分记录支持当前筛选结果摘要
- 当前只开放：
  - 姓名
  - 学号
  - 性别
  - 状态
  - 排序
- 新增学生时当前只开放：
  - 姓名
  - 学号
  - 性别
  - 状态
  - 排序
- 成功后自动刷新学生列表和当前学生详情

## 5. 当前局限

- `status` 底层仍按字符串写入学生表；当前已把旧系统兼容区任务/藏宝阁/双子星结算，以及考勤场次生成与补录入口切到“参与日常”语义，但仍有少量历史命名与边角兼容逻辑待继续收口
- 当前删除策略仍偏保守：只允许删除没有历史积分/考勤且无剩余高风险兼容引用的学生

## 6. 当前结论

学生域已经不再完全只读，至少“新增学生 + 基础资料修正 + 主小组/主宿舍/岗位调整 + 独立状态字典治理 + 安全删除策略”这几条高频切换路径已经具备。后续如果继续扩学生写能力，应优先补：

1. 继续清理剩余历史命名与零散兼容区里的 `active` 旧语义
2. 学生删除策略的真实数据库级回归演练
